## What is DSDS?

DSDS (Design System Documentation Spec) is a JSON-based format for documenting design systems. It gives every piece of documentation ŌĆö components, tokens, themes, foundations, patterns ŌĆö in a machine-readable structure.

<ds-callout>

**Key idea:** DSDS documents the *how and why* of your design system ŌĆö not the token values themselves. It complements the [W3C Design Tokens Format](https://www.w3.org/community/reports/design-tokens/CG-FINAL-format-20251028/) which handles the *what*.

</ds-callout>

### What you get

- **Structured** ŌĆö every section has a defined shape, no guessing
- **Machine-readable** ŌĆö tools can parse, generate, validate, and transform it
- **Portable** ŌĆö not locked to any docs tool or platform
- **Extensible** ŌĆö add vendor metadata without breaking compatibility
- **Validatable** ŌĆö JSON Schema catches errors before they reach consumers

## Document structure

A DSDS file requires `dsdsVersion` and one of two shapes:

### Single-entity files

Use `entity` when each component, token, or pattern lives in its own file. The `kind` field on the entity identifies its type.

```json
{
  "$schema": "https://designsystemdocspec.org/v0.3/dsds.bundled.schema.json",
  "dsdsVersion": "0.3",
  "entity": {
    "kind": "component",
    "identifier": "button",
    "name": "Button",
    "metadata": [
      { "kind": "description", "value": "An interactive element that triggers an action." },
      { "kind": "status", "status": "stable" }
    ],
    "documentBlocks": []
  }
}
```

### Multi-entity files

Use `documentation` when you want to organize multiple entities in one file, or as a manifest that references individual entity files.

```json
{
  "dsdsVersion": "0.3",
  "documentation": [
    {
      "name": "My Design System",
      "components": [
        { "kind": "component", "identifier": "button", "...": "" }
      ],
      "tokenGroups": [
        { "kind": "token-group", "identifier": "color-palette", "...": "" }
      ],
      "themes": [
        { "kind": "theme", "identifier": "dark", "...": "" }
      ]
    }
  ]
}
```

- `dsdsVersion` ŌĆö always `"0.2"` for this version
- `documentation` ŌĆö array of named groups, each with typed entity arrays
- `components`, `tokenGroups`, `themes`, `foundations`, `patterns` ŌĆö separate arrays for each entity type. All are optional.

### Linking files with `$ref`

Entity arrays accept `$ref` objects that point to other DSDS files. This lets you keep entities in individual files while assembling them in a manifest:

```json
{
  "dsdsVersion": "0.3",
  "documentation": [
    {
      "name": "Acme Design System",
      "components": [
        { "$ref": "./button.dsds.json#/entity" },
        { "$ref": "./link.dsds.json#/entity" }
      ]
    }
  ]
}
```

The `#/entity` fragment is a JSON Pointer targeting the `entity` value in the referenced file. Tooling resolves `$ref` objects before validation using a library like [`json-schema-ref-parser`](https://github.com/APIDevTools/json-schema-ref-parser).

<ds-callout variant="tip">

**Tip:** Use single-entity files + a `$ref` manifest for large systems where each component is owned by a different team. Use a single multi-entity file for smaller systems or when you want everything in one place.

</ds-callout>

## Entity types

Every entity has a `kind` field that tells tools what type of thing it is. There are six kinds.

| Kind | Code | Description |
|------|------|-------------|
| component | `"component"` | A reusable UI element ŌĆö buttons, inputs, modals. Supports anatomy, API, variants, states, and design specs. |
| token | `"token"` | A single design token ŌĆö color, spacing, typography. Carries type, category, and usage rules. Link to W3C DTCG definitions via `source`. |
| token-group | `"token-group"` | A nested group of related tokens. Recursive ŌĆö groups can contain groups. |
| theme | `"theme"` | A named set of token overrides ŌĆö dark mode, high-contrast, compact density, brand variants. Lists which token names change; the DTCG file holds the actual values. |
| foundation | `"foundation"` | A design base ŌĆö color, typography, spacing, elevation. Carries principles, scales, and motion definitions. |
| pattern | `"pattern"` | A broad action pattern ŌĆö nav, error messaging, empty states. Describes how components compose to solve a user need. |

### Common properties

Every entity shares these fields:

{/* Hand-rolled: cross-entity summary that intentionally diverges from any single schema. */}

| Property | Required | Description |
|----------|----------|-------------|
| `kind` | Yes | Entity type |
| `identifier` | Yes\* | Machine-readable identifier (`^[a-z][a-z0-9-]*$`, except tokens) |
| `name` | Yes\* | Human-readable label |
| `description` | Yes\* | What this entity is and does (CommonMark supported) |
| `status` | No | `"stable"`, `"experimental"`, `"draft"`, `"deprecated"` ŌĆö or an object with per-platform status |
| `since` | No | Version when this entity first shipped |
| `tags` | No | Array of freeform category tags |
| `documentBlocks` | No | Array of typed document block objects |
| `agents` | No | Agent-optimized context for AI/LLM consumption |

\* Required for component, foundation, and pattern. Token requires `identifier` + `tokenType`.

The optional `agents` object provides context written for AI/LLM consumers ŌĆö `intent` for quick relevance checks, `keywords` for retrieval, `constraints` with enforcement levels and optional `evidence` and inline `examples`, `disambiguation` against similar entities (with optional `entityType`), and `antiPatterns` with corrective alternatives. The `verified` and `verifiedAgainst` fields signal data freshness.

### Status: string or object

For the simple case, status is a string:

```json
{ "status": "stable" }
```

When you need per-platform tracking:

```json
{
  "status": {
    "overall": "stable",
    "platforms": {
      "react": { "status": "stable", "since": "1.0.0" },
      "ios": { "status": "experimental", "since": "3.0.0" }
    }
  }
}
```

## The document block system

Structured docs live in the `documentBlocks` array on each entity. Each document block entry is typed by its `kind` field.

| Kind | Applies To | Description |
|------|-------------|-------------|
| `import` | component | Platform-specific import statements for the component |
| `anatomy` | component, pattern | Named parts that make up the entity, with optional design-token mapping per part |
| `api` | component | Properties, events, slots, and CSS custom properties |
| `events` | component, pattern | Events emitted by the component ŌĆö trigger conditions, return shapes, DOM behavior |
| `variants` | component, pattern | Named variant axes (e.g. size, color) with discrete values |
| `states` | component, pattern | Interactive states (hover, disabled, ŌĆ”) with per-state token overrides |
| `purpose` | all | When to use / when not to use, with options for negative cases |
| `guideline` | all | Actionable guidance paired with rationale and enforcement level |
| `accessibility` | component, pattern | WCAG level, keyboard interactions, ARIA attributes, screen-reader behavior |
| `content` | component, pattern | Writing and labelling guidelines for UI text |
| `interaction` | component, pattern | Step-by-step action flows with optional wireframes |
| `design-specifications` | component, pattern | Concrete measurements, spacing, and sizing values for implementors |
| `scale` | foundation | Named stops along a design scale (e.g. spacing-100ŌĆ”spacing-900) |
| `motion` | foundation | Named animation/transition presets with duration, easing, and purpose |
| `principle` | foundation | Guiding design principles with rationale and examples |

Every document block type also accepts an optional `agents` property for AI/LLM-optimized context scoped to that section. See [Schema Architecture](schema-architecture.html#block-level-agents) for details.

### Purpose: when to use it

The `purpose` guideline lists scenarios. Each use case is `"positive"` (do use) or `"negative"` (don't use, with an option):

```json
{
  "kind": "purpose",
  "useCases": [
    {
      "description": "When the user needs to trigger an action such as submitting a form.",
      "kind": "positive"
    },
    {
      "description": "When the action navigates to a different page.",
      "kind": "negative",
      "alternative": {
        "identifier": "link",
        "rationale": "Links carry native navigation semantics."
      }
    }
  ]
}
```

### Guidelines: how to use it

Each guideline pairs a `guidance` statement with a `rationale` and an enforcement level:

```json
{
  "kind": "guideline",
  "items": [
    {
      "guidance": "Limit each surface to one primary button.",
      "rationale": "Multiple primary buttons dilute visual hierarchy.",
      "kind": "required",
      "category": "visual-design"
    }
  ]
}
```

Enforcement levels: <ds-badge>required</ds-badge> (MUST), <ds-badge>encouraged</ds-badge> (SHOULD), <ds-badge>informational</ds-badge> (MAY, default), <ds-badge>discouraged</ds-badge> (SHOULD NOT), <ds-badge>prohibited</ds-badge> (MUST NOT).

## Validate your document

DSDS ships with a JSON Schema for validating your documents.

### Using the bundled schema

Add `$schema` to get editor autocompletion and inline validation:

```json
{
  "$schema": "https://designsystemdocspec.org/v0.3/dsds.bundled.schema.json",
  "dsdsVersion": "0.3",
  "entity": {
    "kind": "component",
    "identifier": "my-component",
    "name": "My Component",
    "metadata": []
  }
}
```

### Using the CLI

```bash
# Clone the repo
git clone https://github.com/somerandomdude/design-system-documentation-schema.git
cd design-system-documentation-schema
npm install

# Validate the built-in examples
npm run validate

# Validate your own file
npx ajv validate -s spec/schema/dsds.bundled.schema.json -d my-system.dsds.json
```
