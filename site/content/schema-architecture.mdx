## Schema architecture

The DSDS schema uses three directories plus a root schema:

| Directory | Contents |
|---|---|
| `common/` | Shared primitives used across all schemas ‚Äî richText, statusObject, link, example, extensions, metadata, useCases |
| `entities/` | Entity type schemas ‚Äî component, token (including tokenGroup), theme, foundation, pattern, guide |
| `document-blocks/` | Document block type schemas ‚Äî guideline, anatomy, api, events, import, variant, state, design-specifications, accessibility, purpose, scale, principle, motion, content, interaction, section, steps, plus the scoped union (document-blocks.schema.json) |

## Document structure

A DSDS file is a JSON object with a required `dsdsVersion` and exactly one of two primary shapes: a `documentation` array for multi-entity documents, or an `entity` object for single-entity files. Every valid DSDS document must have one or the other.

### Root properties

<ds-prop-table schema="root" def="$root" />

### Documentation groups

Each entry in `documentation` is a named collection of typed entities. Entities are organized into separate typed arrays ‚Äî one for each entity kind. All arrays are optional; include only the ones your documentation needs.

Every entity array accepts either inline entity objects or `$ref` objects pointing to external DSDS files (see [Multi-file documents](#multi-file-documents)). The `documentation` array itself also accepts `$ref` objects, allowing whole groups to be defined in separate files.

<ds-prop-table schema="root" def="documentationGroup" />

### Single-entity documents

When each entity lives in its own file, use the `entity` property instead of `documentation`. The `kind` field on the entity identifies its type.

```json
{
  "$schema": "https://designsystemdocspec.org/v{{VERSION}}/dsds.bundled.schema.json",
  "dsdsVersion": "{{VERSION}}",
  "entity": {
    "kind": "component",
    "identifier": "button",
    "name": "Button",
    "metadata": [
      { "kind": "description", "value": "An interactive element that triggers an action." },
      { "kind": "status", "status": "stable" }
    ],
    "documentBlocks": []
  }
}
```

Valid `kind` values for `entity`: `"component"`, `"token"`, `"token-group"`, `"theme"`, `"foundation"`, `"pattern"`.

### Multi-file documents

Entity arrays support `$ref` JSON Reference objects alongside inline entities. This allows large design systems to split documentation across individual files while maintaining a single manifest that assembles them.

Each `$ref` object has one property: `$ref`. Its value is a URI or relative path to a DSDS file, optionally followed by a JSON Pointer fragment that targets a specific value within that file.

**Individual entity file** (`button.dsds.json`):
```json
{
  "dsdsVersion": "{{VERSION}}",
  "entity": {
    "kind": "component",
    "identifier": "button",
    "name": "Button",
    "metadata": []
  }
}
```

**Manifest file** (`index.dsds.json`):
```json
{
  "dsdsVersion": "{{VERSION}}",
  "documentation": [
    {
      "name": "Acme Design System",
      "components": [
        { "$ref": "./button.dsds.json#/entity" },
        { "$ref": "./link.dsds.json#/entity" }
      ],
      "tokenGroups": [
        { "$ref": "./tokens/color.dsds.json#/entity" }
      ]
    }
  ]
}
```

The `#/entity` fragment is a JSON Pointer ‚Äî it pulls the `entity` value out of the referenced file. Tooling resolves all `$ref` objects into a merged document before validation, using a library such as [`json-schema-ref-parser`](https://github.com/APIDevTools/json-schema-ref-parser). The schema validates the structure of `$ref` objects; resolution and deep validation of referenced content is the responsibility of consuming tooling.

### Entity kind discriminator

Every entity carries a `kind` property ‚Äî a required string const naming the entity type. Entities live in typed arrays (`components`, `tokenGroups`, `themes`, `foundations`, `patterns`), so the array name already pins the expected kind. The `kind` property is still required on each entity for self-description and validation.

<ds-entity-table />

Each typed array accepts only its matching entity type. A documentation group uses as many or as few arrays as needed:

```json
{
  "name": "Acme Design System",
  "components": [
    { "kind": "component", "identifier": "button", "..." }
  ],
  "tokenGroups": [
    { "kind": "token-group", "identifier": "color-palette", "..." }
  ],
  "themes": [
    { "kind": "theme", "identifier": "dark", "..." }
  ],
  "foundations": [
    { "kind": "foundation", "identifier": "spacing", "..." }
  ],
  "patterns": [
    { "kind": "pattern", "identifier": "empty-state", "..." }
  ],
  "guides": [
    { "kind": "guide", "identifier": "getting-started", "..." }
  ]
}
```

### System info

The root `systemInfo` object holds the design system's **identity and provenance** ‚Äî its name, version, organization, URL, and license. This is the system's equivalent of an entity's `identifier` and `name`. It is distinct from the descriptive `metadata` array carried by entities (see [Entity metadata](#entity-metadata)); the two are deliberately separate concepts despite both being "data about" their subject.

<ds-prop-table schema="common/system-info" def="systemInfo" />

### Root purpose

The optional `purpose` property on the root document describes the design system: what it is for, who it serves, and when teams should or should not adopt it. This is a **system-level** property; it applies to the design system as a whole, not to each entity.

{/* Hand-rolled: the root `purpose` object is defined inline in `dsds.schema.json` and has no entry in `$defs`, so it can't be rendered by the shortcode. */}

| Property | Type | Required | Description |
|---|---|---|---|
| `description` | `richText` | No | A high-level description of the design system's purpose and goals. |
| `useCases` | `array` | No | An array of use-case entries describing recommended and discouraged adoption scenarios. |

Each use-case entry has this shape:

<ds-prop-table schema="common/purpose" def="useCase" />

```json
{
  "purpose": {
    "description": "Acme DS provides a unified component library for all customer-facing web products.",
    "useCases": [
      {
        "description": "Building a new customer-facing web application on the Acme platform.",
        "stance": "recommended"
      },
      {
        "description": "Building internal tooling with specialized data-heavy dashboards.",
        "stance": "discouraged",
        "alternative": {
          "identifier": "Acme Admin Kit",
          "rationale": "Admin Kit is optimized for data-dense internal interfaces."
        }
      }
    ]
  }
}
```

### Root guidelines

The optional `guidelines` property on the root document defines cross-cutting rules for the whole design system ‚Äî not for any single entity. These are system-wide conventions like "always use semantic tokens" or "test all components at 200% zoom."

Each `guidelines` entry uses the same shape as a [guideline entry](#guideline-guideline) (`guidelineEntry`):

<ds-prop-table schema="document-blocks/guideline" def="guidelineEntry" />

```json
{
  "guidelines": [
    {
      "guidance": "Always use semantic tokens instead of raw color values.",
      "rationale": "Semantic tokens ensure consistent theming and make future design changes non-breaking.",
      "level": "MUST",
      "category": "development"
    },
    {
      "guidance": "Test all components at 200% browser zoom.",
      "rationale": "WCAG 1.4.4 requires content to remain functional at 200% zoom.",
      "level": "MUST",
      "category": "accessibility",
      "criteria": [
        {
          "url": "https://www.w3.org/TR/WCAG22/#resize-text",
          "label": "WCAG 1.4.4 Resize Text"
        }
      ]
    },
    {
      "guidance": "Do not override token values at the component level.",
      "rationale": "Component-level overrides break theme consistency and make system-wide updates unreliable.",
      "level": "MUST_NOT",
      "category": "development"
    }
  ]
}
```

> **System-level vs. entity-level:** The root `purpose` and `guidelines` properties describe the design system _as a whole_. They differ from the entity-level `purpose` and `guideline` document block types inside each entity's `documentBlocks` array. Entity-level document block entries describe when and how to use a specific component, token, or pattern. System-level properties describe when and how to use the design system itself. Both levels follow the same structure but serve different audiences and scopes.

## Entity types

Every entity carries the same small identity surface ‚Äî a `kind` discriminator, an `identifier`, and (for most kinds) a `name` ‚Äî plus a `metadata` array for descriptive fields and a `documentBlocks` array for all structured docs. Everything beyond identity (description, status, tags, and so on) lives as a typed entry in the `metadata` array rather than as a top-level property (see [Entity metadata](#entity-metadata)). The `documentBlocks` array accepts only the document block types suited to that entity type.

### Common entity properties

These properties appear on every entity type. Token and token-group differ slightly ‚Äî see the [Token](#token) and [Token group](#token-group) sections.

{/* Hand-authored cross-entity summary of the shared entity envelope; spans all entity types rather than mirroring one schema $def. */}

| Property | Type | Required | Notes |
|---|---|---|---|
| `kind` | `string` | Yes | Entity kind discriminator (see [Entity kind discriminator](#entity-kind-discriminator)). |
| `identifier` | `string` | Yes | Machine-readable identifier. Components, foundations, patterns, themes, and guides enforce `^[a-z][a-z0-9-]*$`. Token and token-group identifiers are unconstrained by design (see [Token Identifier Exception](#token-identifier-exception)). |
| `name` | `string` | Varies | Human-readable name. Required on component, foundation, pattern, theme, and guide. **Not present** on token or token-group ‚Äî the identifier itself serves as the display label. |
| `metadata` | `array` | No | Ordered list of typed metadata entries ‚Äî description, status, tags, and more. See [Entity metadata](#entity-metadata). |
| `documentBlocks` | `array` | No | Typed document block objects. See [Document block](#document-block). |
| `agents` | `object` | No | Agent-optimized context for efficient agentic consumption. See [Agents](#agents). |
| `$extensions` | `object` | No | Vendor-specific extensions. See [Extensions](#extensions). |

### Entity metadata

Identity beyond `kind`, `identifier`, and `name` lives in the `metadata` array ‚Äî an ordered list of typed entries, each tagged by its own `kind`. Keeping these out of the entity's top level keeps the entity surface small and lets new metadata kinds be added without changing the entity shape. All entries are optional.

<ds-metadata-kinds-table />

```json
{
  "kind": "foundation",
  "identifier": "spacing",
  "name": "Spacing",
  "metadata": [
    { "kind": "description", "value": "The spatial system governing layout rhythm and density." },
    { "kind": "status", "status": "stable" },
    { "kind": "tags", "items": ["layout", "spacing"] }
  ],
  "documentBlocks": []
}
```

### Component

A reusable UI element. Accepts **component-scoped document block types** (anatomy, api, variants, states, design-specifications) and all general document block types.

Required properties: `kind`, `identifier`, `name`. (`description` and `status` are optional `metadata` entries ‚Äî see [Entity metadata](#entity-metadata).)

### Token

A single design token. Accepts **general document block types** only.

Required properties: `kind`, `identifier`, `tokenType`.

More properties:

<ds-prop-table schema="entities/token" def="token" delta />

DSDS does not duplicate token values or platform identifiers. The W3C Design Tokens Format file is the source of truth for values. Use the `source` property to link tokens to their DTCG definitions.

Tokens have no separate `name` property by design. The token `identifier` itself serves as the human-readable display label. Token identifier conventions (dots, slashes, kebab-case) are already human-readable in their native format.

Like every entity, a token's descriptive fields (`description`, `status`, `tags`, ‚Ä¶) live in its optional `metadata` array ‚Äî see [Entity metadata](#entity-metadata) ‚Äî not as top-level properties. Tokens keep these light to reduce verbosity at scale.

### Token group

A nested group of related tokens. The `children` array can hold token objects, nested token group objects, or a mix of both ‚Äî forming a recursive tree of any depth. Accepts **general document block types** only.

Required properties: `kind`, `identifier`.

More properties:

<ds-prop-table schema="entities/token" def="tokenGroup" delta />

A token group has no `name`; like tokens, its descriptive fields live in the optional `metadata` array. Children inherit `tokenType` from the parent group when they omit their own value.

### Theme

A named set of token overrides that adapt the system to a specific context (color mode, density, brand variant). Accepts **general document block types**.

Required properties: `kind`, `identifier`, `name`. A theme's substance is its `overrides` array (documented below); `description` and `status` are optional `metadata` entries.

More properties:

<ds-prop-table schema="entities/theme" def="theme" delta />

Each `tokenOverride` names a token via `token` (the token identifier) and may carry a `description` explaining the change. Override values live in the DTCG `source` file, not in DSDS.

### Foundation

A broad design foundation governing a domain like color, typography, spacing, elevation, motion, or content. Accepts **foundation-scoped document block types** (principles, scale, motion) and all general document block types.

Required properties: `kind`, `identifier`, `name`. (`description` and `status` are optional `metadata` entries ‚Äî see [Entity metadata](#entity-metadata).)

Foundations document base visual attributes like color and spacing, plus cross-cutting concerns like accessibility, motion systems, and content rules. The `category` property classifies the domain (e.g., `"color"`, `"typography"`, `"spacing"`, `"motion"`, `"accessibility"`, `"content"`).

### Pattern

A broad interaction pattern ‚Äî a recurring, multi-component solution to a common UX problem. Accepts **pattern-scoped document block types** (interactions) and **shared structural document block types** (anatomy, variants, states) as well as all general document block types.

Required properties: `kind`, `identifier`, `name`. (`description` and `status` are optional `metadata` entries ‚Äî see [Entity metadata](#entity-metadata).)

Patterns reference their participating components through the `links` array (with `kind: "component"`, `identifier`, and `role`) instead of through a dedicated document block.

### Guide

A long-form, reading-oriented document ‚Äî a getting-started walkthrough, contribution guide, tutorial, conceptual overview, or migration guide. Guides fill the gap left by the artifact-bound entities (which each document a single component, token, foundation, or pattern): they document a *journey* or a *concept* rather than one thing. Accepts the **guide-scoped** `steps` block, the `import` block, and all general document block types ‚Äî including the free-form `section` block that forms the backbone of a guide.

Required properties: `kind`, `identifier`, `name`. (`description` and `status` are optional `metadata` entries.)

The *kind* of guide (getting-started, contribution, tutorial, concept, migration) is expressed through a `category` metadata entry, keeping the entity flexible enough for any kind of guidance.

## Agents

The optional `agents` property provides context intended exclusively for agentic (AI/LLM) consumption. All properties are optional.

<ds-prop-table schema="common/agents" def="agents" />

```json
{
  "agents": {
    "intent": "Trigger a user action within the current view without causing page navigation.",
    "constraints": [
      { "rule": "Do not use for navigation between pages.", "level": "MUST_NOT" }
    ],
    "disambiguation": [
      { "entity": "link", "distinction": "Use button for in-page actions; use link for navigation to a URL." }
    ],
    "keywords": ["action", "submit", "click", "CTA", "trigger"]
  }
}
```

The `constraints`, `disambiguation`, and `antiPatterns` arrays carry their own entry shapes:

#### Constraint entry

A single must/should rule for AI consumers. `evidence` holds empirical backing (test data, audit findings) and is kept separate from `context` so tools can tell test-derived rules from conventional guidance.

<ds-prop-table schema="common/agents" def="agents" path="constraints.items" />

#### Disambiguation entry

Tells an agent how to tell this artifact apart from a similar one. `entityType` classifies the thing being disambiguated (e.g., another component, a pattern).

<ds-prop-table schema="common/agents" def="agents" path="disambiguation.items" />

#### Anti-pattern entry

<ds-prop-table schema="common/agents" def="agents" path="antiPatterns.items" />

### Block-level agents

Document blocks also accept the same optional `agents` object. Block-level agent context is scoped to that specific documentation section ‚Äî an anatomy block's agents might disambiguate part naming, while a guideline block's agents might provide enforcement constraints for AI code generation.

```json
{
  "kind": "guideline",
  "items": ["..."],
  "agents": {
    "intent": "Enforce correct Button usage in generated UI code.",
    "constraints": [
      { "rule": "Never place two primary buttons on the same surface.", "level": "MUST_NOT" }
    ],
    "keywords": ["button rules", "best practices", "usage guidelines"]
  }
}
```

### Agent context collections

For system-wide agent guidance spanning many entities, use `agent-collection.schema.json`. A collection file contains named `sections`, each validated as a full agents block:

```json
{
  "name": "Pre-Generation Checklist",
  "updated": "2025-07-19",
  "sections": {
    "button": {
      "intent": "Use Button for all interactive action triggers.",
      "constraints": [
        { "rule": "Never override height below 36px.", "level": "MUST", "evidence": "9/10 agent test runs failed." }
      ],
      "verified": "2025-07-19"
    }
  }
}
```

## Status

The `statusObject` groups lifecycle status, per-platform readiness, and deprecation notices into a single object. Every entity that carries status uses this object instead of a bare string.

<ds-prop-table schema="common/status" def="statusObject" />

### Status values

Status values MUST be lowercase kebab-case (pattern: `^[a-z][a-z0-9-]*$`). The four standard values are:

| Value | Meaning |
|---|---|
| `draft` | Under development, not ready for use. |
| `experimental` | Available but API may change without notice. |
| `stable` | Production-ready, changes follow semver. |
| `deprecated` | Scheduled for removal. `deprecationNotice` is required. |

Custom values are permitted (e.g., `"sunset"`, `"archived"`, `"beta"`) and MUST follow the same pattern.

## Document block

Document block entries are the core unit of structured docs in DSDS. Every piece of docs on an entity is a document block object with a `kind` discriminator. Document block types cover:

- guidelines, purpose, accessibility, content, and narrative sections (general ‚Äî all entity types)
- anatomy, API specs, events, variants, states, design specifications, and import (component-scoped)
- principles, scales, and motion definitions (foundation-scoped)
- interactions (pattern-scoped)
- step-by-step procedures (guide-scoped)

### Scoped document block unions

Each entity type accepts only the document block types relevant to it. The scoping rationale:

- **Components** get full structural docs. They are rendered UI elements with code-level interfaces, visual anatomy, configurable dimensions, interactive states, and measurable design specs.
- **Patterns** get structural docs without code-level details ‚Äî they have visual layouts (anatomy), sub-types (variants), and states, but they are not single-component code interfaces. They do not accept `api` or `design-specifications`.
- **Foundations** get domain-level docs ‚Äî they govern a visual or cross-cutting domain through guiding principles, constrained value progressions (scales), and motion definitions.
- **Tokens** get only general document block types ‚Äî they are atomic values with usage guidance but no visual structure or interaction behavior.
- **Guides** get reading-oriented docs ‚Äî step-by-step `steps` procedures plus `import` for setup instructions, and lean on the general `section` block for narrative prose. They carry no structural or measurable specs.

**General** (available on all entity types): guideline, purpose, accessibility, content, and section (free-form narrative prose).

## General document block types

These document block types are available on **all** entity types.

### Guideline (`guideline`)

Documents actionable usage rules for an entity. Each item is a self-contained rule pairing guidance with a rationale. Guidelines tell the reader _how_ to use the entity correctly after choosing it. For guidance on _whether_ to use it at all, see [Purpose](#purpose-purpose).

#### Guideline entry

<ds-prop-table schema="document-blocks/guideline" def="guidelineEntry" />

The `level` field is named for RFC 2119 requirement levels ‚Äî the same value set as `agents.constraints[].level`:

| Value | Meaning |
|---|---|
| `MUST` | Required ‚Äî non-compliance is a defect. |
| `SHOULD` | Follow in most cases; exceptions require justification. |
| `SHOULD_NOT` | Avoid unless justified. |
| `MUST_NOT` | Prohibited ‚Äî violations are defects. |

### Purpose (`purpose`)

Documents what an entity is for ‚Äî the concrete scenarios where it is the right choice and where a different entity fits better. Purpose document block entries help the reader decide _whether_ to reach for this entity before they start using it.

The purpose document block type wraps the common `useCases` data model with a `kind` discriminator. `useCases` is an **array** of scenario entries, each with a `description`, a `stance` of `"recommended"` (when to use) or `"discouraged"` (when to choose something else), and ‚Äî for discouraged cases ‚Äî an `alternative`:

```json
{
  "kind": "purpose",
  "useCases": [
    {
      "stance": "recommended",
      "description": "When the user needs to trigger an action such as submitting a form."
    },
    {
      "stance": "discouraged",
      "description": "When the action navigates to a different page.",
      "alternative": {
        "identifier": "link",
        "rationale": "Links carry native navigation semantics."
      }
    }
  ]
}
```

This is the same `useCases` shape used by the root [`purpose`](#root-purpose) property (full model in [Use Cases](#use-cases)).

### Accessibility (`accessibility`)

Documents structured accessibility specs. It captures _machine-readable specs_ for:

- keyboard interactions
- ARIA attributes
- screen reader behavior
- focus management
- color contrast pairs
- motion concerns

For actionable accessibility _rules_ with rationale (e.g., "Provide an aria-label for icon-only buttons"), use a guideline entry with `category: "accessibility"` instead.

<ds-prop-table schema="document-blocks/accessibility" def="accessibility" />

### Examples (embedded, not a standalone block)

DSDS has no standalone `examples` document block. Instead, the reusable `example` model is **embedded** inside other blocks ‚Äî most commonly the `examples` array on a [guideline entry](#guideline-guideline) and on internal success criteria, and the `examples` array on a guide's `section` and `steps` entries.

Each `example` requires at least a `presentation` or a `value` (or both):

- **Presentation**: A visual or interactive demo ‚Äî one of `presentationImage`, `presentationVideo`, `presentationCode`, or `presentationUrl`.
- **Value**: A literal value for API property examples (e.g., `"primary"`, `44`, `true`). When given without a presentation, the example is a concrete data point.

Optional `title` and `description` provide context.

### Content (`content`)

Documents structured content documentation ‚Äî recommended action labels and localization/i18n concerns. Content document block entries complement guideline document block entries: guidelines say "Use sentence case" (a _rule_), content entries say "Add: Takes an existing object and uses it in a new context" (a _reference_).

At least one of `labels` or `localization` must be present.

#### Content label entry

<ds-prop-table schema="document-blocks/content" def="contentLabelEntry" />

#### Localization entry

<ds-prop-table schema="document-blocks/content" def="localizationEntry" />

### Section (`section`)

Free-form narrative prose organized into titled, optionally nested sections ‚Äî the backbone of conceptual overviews, explanations, rationale, and any reading-oriented content the structured block kinds do not capture. Available on every entity type. Each `sectionEntry` pairs a `title` with an optional rich-text `body`, illustrative `examples`, related `links`, and recursive `sections` for a heading hierarchy. Reach for a structured block kind first; use `section` only for what they cannot express.

#### Section entry

## Component-scoped document block types

These document block types are centered on **component** entities. `api` and `design-specifications` are component-only; `anatomy`, `variants`, `states`, and `events` are shared with patterns; and `import` is shared with guides (as noted in the [scope table](#scoped-document-block-unions)).

### Import (`import`)

How to install and import the artifact across platforms ‚Äî package names, import statements, and setup context. Each item documents one platform.

#### Import entry

<ds-prop-table schema="document-blocks/import" def="importEntry" />

### Anatomy (`anatomy`)

Documents the visual structure of a component or pattern by listing its named sub-elements (parts). Each part can reference the design tokens that style it, bridging the entity's visual design and its token architecture.

For components, parts represent rendered UI sub-elements (container, label, icon, focus-ring). For patterns, parts represent the structural sections of the pattern's visual layout (image, title, body, primary action).

#### Anatomy entry

<ds-prop-table schema="document-blocks/anatomy" def="anatomyEntry" />

### API (`api`)

Documents the code-level interface of a component on a single platform. For multi-platform components, create one API block per platform using the `platform` property.

Sections:

- `properties` (apiProperty[])
- `events` (apiEvent[])
- `slots` (apiSlot[])
- `cssCustomProperties` (apiCssCustomProperty[])
- `cssParts` (apiCssPart[])
- `dataAttributes` (apiDataAttribute[])
- `methods` (apiMethod[])

#### Property entry (`apiProperty`)

<ds-prop-table schema="document-blocks/api" def="apiProperty" />

For enum properties, set `type` to the base type (`"string"`) and list the accepted values in `values`:

```json
{
  "identifier": "variant",
  "type": "string",
  "values": ["default", "primary", "ghost", "danger"],
  "description": "The visual style of the button.",
  "required": false,
  "defaultValue": "default"
}
```

#### Event entry (`apiEvent`)

<ds-prop-table schema="document-blocks/api" def="apiEvent" />

### Events (`events`)

A standalone block documenting the events an artifact emits ‚Äî trigger conditions, return shapes, DOM behavior (bubbling, cancelability, composed), and lifecycle metadata. Shared by components and patterns. It is richer than the `api` block's inline `events` array: use `api.events` for a quick interface listing, and the `events` block when an event's behavior warrants its own structured entry.

#### Event entry (`eventEntry`)

<ds-prop-table schema="document-blocks/events" def="eventEntry" />

### Variants (`variants`)

Documents all dimensions of visual or behavioral variation. Each item is a single axis of config (e.g., `"size"`, `"emphasis"`, `"full-width"`). Multiple axes document orthogonal dimensions that combine independently.

The optional `exclusions` array documents invalid combinations across dimensions:

```json
{
  "kind": "variants",
  "items": [
    { "identifier": "emphasis", "description": "...", "values": [...] },
    { "identifier": "size", "description": "...", "values": [...] }
  ],
  "exclusions": [
    {
      "conditions": [
        { "dimension": "emphasis", "value": "ghost" },
        { "dimension": "size", "value": "sm" }
      ],
      "description": "Ghost emphasis at small size does not provide adequate visual affordance."
    }
  ]
}
```

Each exclusion requires at least two conditions. A single condition would exclude a whole value, which should be removed from the dimension's values array instead.

#### Variant value

<ds-prop-table schema="document-blocks/variant" def="variantValue" />

### States (`states`)

Documents all interactive states ‚Äî hover, focus, active, disabled, loading, selected, error, etc. ‚Äî with triggers, token overrides, and previews.

#### State entry

<ds-prop-table schema="document-blocks/state" def="stateEntry" />

### Design specifications (`design-specifications`)

Documents measurable visual specs ‚Äî token inventory, spacing relationships, dimension constraints, typography settings, and responsive behavior. At least one section must be present.

<ds-prop-table schema="document-blocks/design-specifications" def="designSpecifications" />

#### Variants, sizes, and states

The base-level `properties` map defines the default spec ‚Äî typically the primary variant at medium size in the default state. Values are always strings: either design token names (e.g., `"button-background"`, `"space-4"`) or raw CSS values (e.g., `"#0055b3"`, `"16px"`, `"transparent"`). Systems that do not use tokens provide raw values directly.

The remaining arrays group design properties by condition:

- **`variants`** ‚Äî by variant. Each entry has an `identifier` and its own `properties` map, plus optional `spacing`, `sizing`, and `typography` sections.
- **`sizes`** ‚Äî by size, including size-specific spacing, sizing, and typography.
- **`states`** ‚Äî by interactive state (e.g., hover, focus, disabled).

The `variantStates` array handles variant√óstate combinations where the state's visual treatment differs across variants. Each entry specifies both a `variant` and a `state` along with the `properties` map for that combination.

Each entry is self-contained ‚Äî it holds the complete set of values for that condition, not a diff of the base properties. For example, a `"danger"` variant entry carries its own token names (like `"button-danger-bg"`) or raw values, independent of the base `properties`.

Note: the `tokens` field on state entries (in the states block) is a descriptive summary for designers ‚Äî it shows *which* tokens change in a given state. The authoritative resolved values for each variant √ó state combination live in the `design-specifications` block's `variants`, `states`, and `variantStates` arrays.

## Foundation-scoped document block types

These document block types are only accepted on **foundation** entities.

### Principles (`principles`)

Documents high-level guiding principles that shape decisions for a domain. Principles answer "what do we believe?" and "what constraints do we accept?" ‚Äî they sit above specific best practices.

Each `principleEntry` has a `title` (short, memorable name) and a `description` (what it means in practice).

### Scale (`scale`)

Documents an ordered progression of values forming a visual scale (type scale, spacing scale, elevation scale, or similar).

Each `scaleStep` references a design token and optionally provides a `label`, `value` (display convenience), and `description` (usage notes). Steps are ordered from smallest/lowest to largest/highest.

The scale block requires an `identifier`, `description`, and `steps` array.

### Motion (`motion`)

Documents the motion system ‚Äî named easing curves, their cubic-bezier timing functions, recommended durations, and usage guidance. Each item is a single easing definition that designers and engineers select from when adding animation or transitions.

## Links

The `links` array provides typed references to external resources and internal DSDS entity relationships. Links combine external URL references and internal entity references into a single model.

Every link requires a `kind` and at least one of `url` (for external resources) or `identifier` (for internal DSDS entity references). Both can be present when an internal entity also has an addressable URL.

<ds-prop-table schema="common/link" def="link" />

### Standard link types

**External resource types:** `"source"`, `"design"`, `"storybook"`, `"documentation"`, `"package"`, `"repository"`.

**Relationship types:** `"alternative"`, `"parent"`, `"child"`, `"related"`.

## Use cases

The `useCases` data model provides scenario-driven guidance for when something is ‚Äî and is not ‚Äî the right choice. It appears in two places, with an identical shape in both:

1. **The root [`purpose`](#root-purpose) property** ‚Äî adoption guidance for the design system as a whole.
2. **The [`purpose`](#purpose-purpose) document block** ‚Äî when to reach for a specific entity.

In both, `useCases` is an **array** of scenario entries. Each entry has a `description` and a `stance` of `"recommended"` (when to use) or `"discouraged"` (when to choose something else); discouraged entries SHOULD include an `alternative` pointing to a more fitting entity:

```json
[
  {
    "stance": "recommended",
    "description": "Scenario where this is the right choice."
  },
  {
    "stance": "discouraged",
    "description": "Scenario where something else fits better.",
    "alternative": {
      "identifier": "alternative-artifact",
      "rationale": "Why the alternative is better."
    }
  }
]
```

## Extends

The `extends` mechanism enables **systems-of-systems** architectures ‚Äî where an extension design system inherits from a core system. It operates at two levels:

### Document-level extends (`documentExtends`)

The root `extends` property declares that the whole DSDS document inherits from another DSDS document. This sets up the parent‚Äìchild relationship between systems.

<ds-prop-table schema="common/extends" def="documentExtends" />

```json
{
  "extends": {
    "system": "Acme Core Design System",
    "url": "https://design.acme.com/v2/core.dsds.json",
    "version": "2.0.0",
    "description": "Adds enterprise-specific components, stricter accessibility requirements, and product-specific token overrides."
  }
}
```

### Entity-level extends (`entityExtends`)

An entity can declare that it extends a base entity from a parent system. This is carried as an `extends` entry in the entity's `metadata` array (`{ "kind": "extends", "extends": { ‚Ä¶ } }`) ‚Äî like every other descriptive field, it lives in `metadata`, not as a top-level property. The `extends` value has this shape:

<ds-prop-table schema="common/extends" def="entityExtends" />

{/* Hand-rolled: the `modifications` array items are defined inline in `common/extends.schema.json` (no $defs entry), so they can't be rendered by the shortcode. */}

Each `modifications` entry has this shape:

| Property | Type | Required | Description |
|---|---|---|---|
| `changeType` | `string` | Yes | `"added"`, `"modified"`, `"removed"`, or `"inherited"`. |
| `target` | `string` | No | What was changed ‚Äî a property name, variant name, guideline, etc. (e.g., `"variant:enterprise"`, `"prop:theme"`, `"guideline:accessibility"`). |
| `description` | `richText` | Yes | A human-readable description of the change. |

```json
{
  "kind": "component",
  "identifier": "button",
  "name": "Button",
  "metadata": [
    { "kind": "description", "value": "Enterprise Button extends core Button with additional variants and props." },
    { "kind": "status", "status": "stable" },
    {
      "kind": "extends",
      "extends": {
        "identifier": "button",
        "system": "Acme Core Design System",
        "version": "2.0.0",
        "description": "Adds an 'enterprise' variant and a 'theme' prop for sub-brand theming.",
        "modifications": [
          { "changeType": "added", "target": "variant:enterprise", "description": "High-emphasis variant using enterprise brand color." },
          { "changeType": "added", "target": "prop:theme", "description": "Prop accepting 'default' or 'admin' for sub-brand palette switching." },
          { "changeType": "inherited", "target": "guideline:anatomy", "description": "Anatomy inherited from core without changes." }
        ]
      }
    }
  ]
}
```

> **Merge semantics are tooling's responsibility.** The `extends` declaration defines the _relationship_ between systems and entities. How properties, guidelines, and tokens are merged, overridden, or inherited is up to consuming tooling ‚Äî the schema does not prescribe resolution rules. The `modifications` array provides a human-readable changelog that tooling MAY use for diff views, migration guides, or docs generation.

See [`examples/extension-system.dsds.json`](examples/extension-system.dsds.json) for a complete enterprise extension system showing both document-level and entity-level extends with modifications.

## Extensions

The root document, each documentation group, and each entity can carry a `$extensions` property. Keys MUST use vendor-specific namespaces (reverse domain name notation recommended). Tools that do not recognize an extension MUST preserve it. Extension data SHOULD NOT duplicate info already in core schema fields.

```json
{
  "$extensions": {
    "com.designTool": {
      "componentId": "1234:5678"
    },
    "com.storybook": {
      "storyId": "components-button--primary"
    }
  }
}
```

## Naming conventions

### Token identifier exception

Most entity types enforce a strict `^[a-z][a-z0-9-]*$` pattern on the `identifier` property. Tokens and token groups are the exception by design. Their identifiers are unconstrained to fit the W3C Design Tokens Format Module, design tool variable systems, and existing token setups. These setups may use dots (`color.text.primary`), slashes (`color/text/primary`), or other separators. Token identifiers _SHOULD_ still be lowercase and human-readable, but the schema does not enforce a pattern.

### Document block type naming

Document block type values follow two naming patterns based on their structural role:

- **Plural names** for document block types that wrap a list of like items in an `items` array: `"variants"`, `"states"`, `"principles"`, `"interactions"`, `"steps"`, `"motion"`.
- **Singular names** for document block types that are self-contained with their own internal structure:
  - `"scale"` (has `identifier`, `steps`)
  - `"anatomy"` (has `parts`)
  - `"api"` (has `properties`, `events`, etc.)
  - `"accessibility"` (has `keyboardInteraction`, `ariaAttributes`, etc.)
  - `"design-specifications"` (has `tokens`, `spacing`, etc.)
  - `"purpose"` (has `useCases`)

The distinction: plural types are containers of interchangeable items where the container has no identity beyond its type. Singular types have meaningful internal structure where properties are named and semantically distinct. Note: `"guideline"` and `"section"` use singular names despite wrapping an `items` list, because each describes a concept (providing guidelines; a body of narrative sections) rather than being a generic plural container.
